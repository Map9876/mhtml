## 知見

- **コーディング知見**: [docs/knowhow/sanitize-and-agent-patterns.md](../../docs/knowhow/sanitize-and-agent-patterns.md) に「Fastify インメモリ LRU キャッシュ」節を追加
- **タスク総括**:
  - **キャッシュキーの区切り文字選定** はセキュリティ境界。NULL byte は「URL や lang に通常含まれない」「衝突を絶対に起こさない」点で最強の選択肢。半角スペースだと `lang="ja en"` のような値で偽ヒットが発生
  - **`Error` のキャッシュ問題**: Error インスタンスを直接 `JSON.stringify` すると `{}` になる。HIT 時と MISS 時でレスポンス構造が変わる隠れバグになりうる。MISS 経路でも同じ正規化を通すことで構造の一貫性を担保
  - **LRU の TTL 設計**: コンストラクタの `ttl` は「全エントリのデフォルト」と誤読されやすい。各 `set()` で TTL を指定する方針なら、コンストラクタからは省く方が意図が明確
  - **エディタの NULL byte 表示**: ソースに NULL byte を直接書くと表示が壊れる。`\0` エスケープシーケンスで書く（ただし Edit ツールが NULL を生のまま入れることがあるので注意、必要なら sed で確認）
